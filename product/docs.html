Documentation

Everything needed to wire up contacts, send SMS under your own sender ID, connect a payment provider to your USSD menus, and receive delivery events on your own backend.

01 Contacts & database connections

Add recipients manually, import a CSV, or keep a live sync from your own database.

MANUAL / CSV
Go to Contacts → Add contact to enter a single number, or Contacts → Import to upload a CSV. Required columns: phone. Optional: first_name, last_name, and any custom field you want available as a merge tag in messages, e.g. {{first_name}}.
CONNECT YOUR DATABASE
Go to Contacts → Data sources → Connect database. Supported: PostgreSQL, MySQL. Enter a read-only connection URI and pick the table + column mapping — Tunnelo SU only ever runs SELECT against this connection, never writes to it.
// Connection URI format
postgres://readonly_user:password@db.yourhost.com:5432/yourdb?sslmode=require
Note — use a dedicated read-only DB user, not your app's main credentials. Tunnelo SU never needs write access to your database.
SYNC BEHAVIOR
Once connected, pick a sync interval (15 min – 24 hr) or trigger Sync now manually. New rows matching your query become contacts automatically; removed rows are marked inactive, not deleted, so historical delivery logs stay intact.

02 Sender ID & sending SMS

Register the name recipients see instead of a phone number, then send.

STEP 1 — SUBMIT YOUR SENDER ID
Go to Settings → Sender IDs → Request new. Enter a name up to 11 alphanumeric characters (e.g. YOURBRAND) and a short reason for use. Submissions are reviewed by the network operators — approval typically takes 24–48 hours. Status shows as pending, approved, or rejected on the same page.
STEP 2 — COMPOSE & SEND
From Send SMS, choose recipients (single number, a group, or an uploaded CSV), pick an approved sender ID from the dropdown, and write your message. Choose Send now or Schedule for a future date/time.
API SEND
Sending also works from your own backend using your API key from Settings → API Keys.
curl -X POST https://api.tunnelo.click/v1/sms/send \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "sender_id": "YOURBRAND",
    "to": ["+233201234567"],
    "message": "Your order has shipped."
  }'

03 Groups & table selection

Organize contacts into reusable lists for targeted sends.

CREATE A GROUP
Go to Contacts → Groups → New group, name it, then add members from existing contacts, a fresh CSV upload, or a connected database table.
SELECT A TABLE FOR A GROUP
With a database connected (see §01), open the group and choose Source → Database table. Pick the table, then map its columns to phone and any merge-tag fields. You can add a WHERE filter so the group only includes rows matching a condition, e.g. plan = 'active'.
USING GROUPS IN A SEND
In Send SMS, select Recipients → Group and pick one or more groups. Duplicate numbers across groups are sent to once per campaign.

04 Connect your payment provider (USSD)

Link a mobile money / payment provider so USSD payment nodes can collect money through your own merchant account.

STEP 1 — ADD MERCHANT CREDENTIALS
Go to Settings → Payment Integration and enter your Merchant ID and API Key from your payment provider. These are used on every payment node across all your USSD menus.
STEP 2 — CUSTOM API (BRING YOUR OWN PROVIDER)
If your provider isn't in the built-in list, choose Provider → Custom API and supply your own charge endpoint plus the request/response field mapping so Tunnelo SU knows how to trigger a charge and read back status.
// Custom provider config
{
  "charge_url": "https://provider.example.com/charge",
  "auth_header": "Authorization: Bearer {api_key}",
  "request_map": { "amount": "amount", "phone": "msisdn" },
  "response_map": { "status": "result.status", "reference": "result.txnId" }
}
STEP 3 — TEST THE CONNECTION
Click Test charge to run a 1-unit test transaction against your own credentials before attaching the provider to a live payment node. A successful test shows status: success with a returned reference.

05 Set up a USSD menu

Build the menu tree visually, then deploy it to your shortcode.

STEP 1 — CREATE THE MENU
Go to USSD → New menu, enter your registered shortcode (e.g. *789#), and name the menu.
STEP 2 — ADD NODES
Add options and pick a type for each: message (display text and end or return to the menu), submenu (opens a nested set of options), or payment (collects an amount and charges it via the provider from §04 — set a prompt, confirmation text, and success/failure responses).
Node typePurpose
messageShow static or dynamic text, then exit or return up a level
submenuNest further numbered options under this one
paymentPrompt for/collect an amount, charge via your merchant account
STEP 3 — DEPLOY
Click Save, then Deploy. The menu goes live on your shortcode immediately — no code, no config files, no redeploy delay.

06 Set up webhooks

Receive delivery status and USSD payment events on your own endpoint in real time.

STEP 1 — REGISTER AN ENDPOINT
Go to Settings → Webhooks → Add endpoint, enter a public HTTPS URL, and choose which events to receive: sms.delivered, sms.failed, ussd.payment.success, ussd.payment.failed.
STEP 2 — VERIFY THE SIGNATURE
Every request includes an X-Tunnelo-Signature header — an HMAC-SHA256 of the raw request body using the signing secret shown on the endpoint's settings page. Reject any request whose computed signature doesn't match.
// Example payload — sms.delivered
{
  "event": "sms.delivered",
  "message_id": "msg_8f2a1c",
  "to": "+233201234567",
  "status": "delivered",
  "timestamp": "2026-07-03T22:10:04Z"
}
STEP 3 — RESPOND CORRECTLY
Return a 2xx status within 5 seconds to acknowledge receipt. Non-2xx or timeouts are retried with backoff up to 5 times; after that the event is marked undelivered in Settings → Webhooks → Logs, where you can inspect and manually replay it.
Tip — keep the handler fast and idempotent (use message_id / reference to dedupe) since retries can deliver the same event more than once.
